## 5. Case Study Framework (Strict)

Every case study must answer these questions. No exceptions. This is not a timeline. It is not a Figma dump. It is a judgment narrative.

### Required Sections

**1. Context & Stakes**
- Who was the user? (Be specific—not "users," but who specifically)
- Why did failure matter? What did it cost?
- What was the business or operational reality?

**2. System Complexity**
- What were the states, rules, constraints, dependencies?
- What was the hidden structure under the visible problem?
- What was technically or organizationally hard?

**3. Key Insight**
- What assumption broke?
- What did you learn that changed the direction?
- This is NOT research theater. One honest insight beats five user interviews listed.

**4. Design Strategy**
- What patterns were chosen and why?
- What did you intentionally NOT do? (This matters.)
- How was complexity managed without removing power?

**5. Outcome**
- What changed? What scaled? What shipped?
- Use real numbers where they exist
- Be honest about what didn't work

**6. Reflection**
- What would you do differently?
- What's still unresolved?

### Rules for Case Studies
- Fewer projects, deeper stories. 3–5 total. Not 10.
- Lead with the most interesting truth, not the safest framing
- If a section doesn't show judgment, it's incomplete
- Diagrams over mockups when possible
- Visual assets should communicate insight at a glance, not document comprehensiveness
- V1/V2 sequencing principle: show polished outcome first, then reveal the chaos—it lands harder

## 7. MLP Case Study — Active Workstream Detail

This is the current build focus. Key decisions already made:

**Narrative Structure:**
- Hero → V2 homepage scroll → The Fire Drill (before/after) → The Stop → Design Strategy → Data Architecture
- V2 shown first (Section 2) so the V1 chaos reveal in Section 3 lands harder
- V1 (dark navy/orange) was not Caleb's design — developed during a "fire drill" by others
- V2 (light, refined) is Caleb's actual work

**Design System:**
- 8 color ramps, 96 colors total
- Pickleball-themed palette names: "Dink Night," "Match Point," "Warm Bounce," etc.
- Named palettes are a differentiator — give them visual prominence

**Navigation Assets:**
- Desktop component state inventory: 5 states
- Mobile nav system view
- Positioned after "The Fire Drill" section as evidence of systems thinking

**Data Architecture:**
- Three-layer schema: Identity → Competition → Output
- Scoring cascade as core mechanic
- Two-part visual approach: relationship diagram + annotated Player Detail UI screenshot
- Event states component: one component, five configurations from single `events.status` field
- This is the strongest technical design story in the case study

**Visual/Diagram Principles for MLP:**
- Diagrams communicate design insight, not database documentation
- Multiple ERD-style iterations were rejected for feeling too technical
- Annotated UI screenshot: Caleb handles annotation layer; base relationship diagram needed

## 12. Working Instructions for Any Agent or Collaborator

### Collaboration Style
- Work in focused chunks: propose → refine → sharpen
- Push back on generic ideas — vagueness is always a problem
- Pressure-test positioning constantly
- End every major response with a clear next step or question

### You Must
- Edit aggressively
- Call out anything that sounds like it could apply to 50 other designers
- Offer sharper alternatives, not just critiques
- Flag anything safe, forgettable, or inspirational-poster-esque
- Prioritize insight over process documentation in all case study writing

### You Must Not
- Add buzzwords
- Recommend "passionate about design" framing
- Suggest timeline-style case study narratives
- Propose UI-execution positioning
- Inflate Tier 2 projects to Tier 1 status
- Write case study sections that document deliverables without demonstrating judgment
- Treat Fre.sh docs as current positioning

### When In Doubt, Ask:
1. Does this reinforce that Caleb turns ambiguity into something teams can trust and ship?
2. Does this show judgment, or just output?
3. Would this headline or copy apply to 50 other designers? If yes, rewrite it.
4. Is this a diagram that communicates insight, or documentation that proves thoroughness?
